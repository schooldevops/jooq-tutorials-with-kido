# Chapter 01: Why jOOQ? 강의 계획서

## 1. 컬리큘럼 제목 분석
* **제목:** Why jOOQ? (JPA, MyBatis와의 비교 분석 및 jOOQ의 Type-safety 철학 이해)
* **분석 내용:**
  * 기존 데이터베이스 액세스 프레임워크인 JPA(객체 관계 매핑)와 MyBatis(SQL 매퍼)의 특징과 한계를 짚어봅니다.
  * jOOQ가 왜 등장하게 되었는지, 그리고 Type-safety(타입 안정성)가 컴파일 타임에 오류를 어떻게 방지해 주는지에 대한 핵심 철학을 학습합니다.
  * 단순한 이론 설명에 그치지 않고, 코드 레벨에서 각각의 방식이 어떻게 차이 나는지 시연하여 실질적인 이해를 돕습니다.

## 2. 강의 계획 작성

### 2.1. 학습 목표
1. **JPA, MyBatis, jOOQ의 차이점 및 각각의 장단점을 명확히 설명할 수 있다.**
2. **jOOQ의 핵심 철학인 'Type-safety'의 의미와 실무적 이점을 이해한다.**
3. **다양한 프레임워크에서의 SQL 작성 흐름을 비교하여 jOOQ의 도입 필요성을 판단할 수 있다.**

### 2.2. 학습 내용
1. **도입:** 현대 자바/코틀린 백엔드 생태계에서의 데이터베이스 액세스 전략 개요.
2. **비교 분석:** 
   * **MyBatis:** XML 기반, 유연하지만 문자열 오타에 취약 (런타임 에러).
   * **JPA (Hibernate):** 객체 중심, 쿼리 복잡도 증가 시 N+1 문제 및 QueryDSL 의존성.
   * **jOOQ:** Database-first 엔진, 스키마 기반 코드 생성을 통한 완벽한 컴파일 타임 타입 검사.
3. **jOOQ의 Type-safety 철학 이해:** DB 스키마가 변경되었을 때 jOOQ 코드가 어떻게 미리 컴파일 에러를 뱉어서 장애를 예방하는지 실습.
4. **시각화 (Mermaid 활용):** 
   * 각 프레임워크의 SQL 실행 흐름을 나타내는 BPMN / 시퀀스 다이어그램 제공.

### 2.3. 기술 선택 및 개발환경 구성 (시연용)
* **언어:** Java 17+, Kotlin 1.9+
* **프레임워크:** Spring Boot 3.x
* **데이터베이스:** PostgreSQL (Docker Compose 활용)
* **도구:** Gradle (jOOQ 코드 제너레이터 플러그인 포함)

### 2.4. 실제 개발 및 테스트 단계 (강의 시연 항목)
1. **간단한 테이블 생성 (users)**
2. MyBatis로 `users` 테이블 조회 코드 작성 및 오타로 인한 런타임 에러 시연.
3. JPA로 동일 내용 작성 시연.
4. jOOQ 코드로 동일 내용 작성 상황에서 필드명 오타 시 컴파일 에러가 나는 짜릿한(?) 경험 제공.
